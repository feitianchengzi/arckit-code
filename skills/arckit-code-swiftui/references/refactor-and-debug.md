# Refactor And Debug

## 重构原则

重构前确认目标：行为保持、架构边界调整、模块迁移、状态收敛、性能治理、测试补强或为新功能铺路。不要把“看起来更整洁”作为单独目标。

分步执行：

1. 锁定当前行为：测试、截图、手测路径、样本数据或日志。
2. 找到最窄边界：View、ViewModel/store、service、router、adapter、model 或 target。
3. 建兼容层或过渡接口，先让调用方稳定。
4. 小步迁移，每步可构建、可回归。
5. 删除旧路径前确认没有调用方、数据迁移和外部入口遗漏。

## 高风险迁移

- 路由重构保留 deeplink/Universal Link/Widget/Push/分享入口 round-trip 样本。
- 状态重构覆盖 loading、error、refresh、cancel、快速重复操作。
- 数据迁移覆盖旧版本、空数据、损坏数据和回滚策略。
- service 重构覆盖错误映射、取消、超时和日志脱敏。

## Debug 循环

1. 区分现象和解释，根因必须等待证据支持。
2. 建立事实链：输入、状态、缓存、请求、存储、派生结果、展示和生命周期。
3. 识别身份和新鲜度：同一资源可能 id 没变但内容、时间戳、缓存层级或完整度已变。
4. 关注边界和时序：Task 取消、重试、缓存命中、权限回调、scene phase、View 重建、并发刷新。
5. 设计可证伪观察点：日志、测试、trace、请求响应、状态快照、截图或 Instruments。
6. 做最小修复，只改证据指向的根因路径和必要防护。
7. 回到用户原始路径验证。

## iOS 常见问题面

SwiftUI 状态未刷新或刷新过度、`.task(id:)` 身份错误、旧任务写回、`@Query` 和缓存身份不一致、权限配置缺失、deeplink 冷热启动差异、二维码/分享文件在目标 App 中被压缩、设备/系统版本/Dynamic Type/网络和生命周期差异。
