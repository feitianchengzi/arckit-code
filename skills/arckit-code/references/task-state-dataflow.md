# State Dataflow 任务规则

## 目标

确定状态归属、数据流和层边界，避免状态多源复制、ViewModel 膨胀、Service 污染 UI 类型和副作用散落。

## 决策规则

- 区分持久化领域数据、非持久化业务状态、局部 UI 状态和系统环境值。
- 每个关键状态只有一个事实来源。
- Model 表达领域数据，Service 表达外部能力，View 负责协调和渲染。
- 派生 display model 必须能响应源数据完整度变化。
- 异步副作用不放在渲染函数或纯展示组件中。

## 反模式

- 为一个局部开关引入全局 observable。
- Service 接口泄漏平台 UI 类型。
- 缓存快照、fallback 数据和完整数据无法表达新鲜度差异。
- 多个字段同时表达同一业务状态。

## 验证

关键状态转换、错误恢复、异步补全和缓存刷新路径应有测试或可复现样本。
