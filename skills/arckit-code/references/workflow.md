# ArcKit Code 写代码工作流

## 基本顺序

1. 读取代码事实，不先套模板。
2. 明确用户要的是实现、修复、审查、重构、补测试还是验证。
3. 判断代码问题类型和技术栈。
4. 读取最小 reference 组合。
5. 先理解现有模式，再修改代码。
6. 保持改动范围和用户目标一致。
7. 验证并记录无法验证的原因。

## 任务事实

编码前至少确认：

- 入口文件、调用链和相关测试。
- 项目使用的包管理器、构建脚本和测试命令。
- 同类功能的目录、命名、错误处理和依赖注入方式。
- 目标平台和运行约束。
- 用户明确要求的不可变边界。

## 修改原则

- 沿用项目现有框架和本地 helper。
- 共享逻辑只有在减少真实复杂度或符合现有模式时才抽象。
- 不为小改动重排目录、迁移架构或引入新依赖。
- 不把平台类型泄漏到通用领域层，除非项目既有边界已经如此且本次不适合重构。
- 涉及数据、缓存、迁移和发布相关配置代码时，优先做可回归验证。

## 输出要求

最终说明包含：

- 改动范围。
- 关键行为变化。
- 已执行验证。
- 未验证项和剩余风险。
